# SPEC-Core Architecture - O (SCAO)

> **Pronunciation:** スペックコア・アーキテクチャ・バイ・オー

本リポジトリは、人間（Spec Author / Owner）が定義する『SPEC（仕様）』をすべての中心（Core）に据え、4つの生成AI（Gemini・ChatGPT・Cursor・Claude）にそれぞれ異なる独立した責務を与えて統治・制御する、堅牢開発フレームワーク「SPEC-Core Architecture - O（SCAO）」の統合テンプレートです。

SCAOの「O」は「Oikawa」の頭文字です。

本リポジトリ自体は、アプリケーション開発用のソースコードリポジトリではありません。実務プロジェクトへそのまま複製して使用するスターターテンプレートでもありません。生成AIを実務レベルで活用するための考え方、責務分離、設定例、運用手順を整理した、**ドキュメント中心の学習教材リポジトリ**です。

学習の目的は次のとおりです。

- 人間が定義したSPECを正本とします。
- 複数の生成AIへ適切な責務を割り当てます。
- 精度の高い回答と成果物を得るため、設計、実装、テスト、レビュー、再設計のサイクルを継続的に回します。
- 無駄な作業、確認漏れ、仕様逸脱、不要な実装、セキュリティ上の見落としを抑制します。

AIの浅い知見や逸脱の発生リスクを低減するため、実務経験に基づく「安全第一（Safety-First）」の設計思想を具現化するために構築しました。小規模の開発現場でも利用でき、初学者の学習用にも利用できることを意識して作成しました。

### 開発サイクル

```text
  ↓

  ↓

  ↓

  ↓

  ↓

```

### AIと人間の責務分担（概要）

| 担当 | 主な責務 |
| --- | --- |
| Gemini | 最新情報調査、公式仕様確認、比較、エラーログ解析 |
| ChatGPT | 要件定義、設計、技術選定、Cursor向け実装指示書作成 |
| Cursor | 指示書に基づく実装、テスト、ローカル検証、差分確認 |
| Claude | 論理矛盾、仕様不一致、バグ、セキュリティ、DRY違反の監査 |
| 人間 | SPEC承認、差戻し判断、最終意思決定 |

## 1. 開発思想：Back To The Basic
AIに仕様から実装まで一任するだけでは、品質と安全の担保が難しい場面がある。AIのエージェント化が進む現代だからこそ、人間が厳格な法（SPEC）を敷き、テストと査読という監査システムで外側を囲う「開発の王道（Basic）」が必要だと考えます。（※あくまでも2026年時点の判断）
本フレームワークでは、AIを「SPECという法律を厳格に守らせる協力者」として、責務分離と監査により統制することを目的としています。

## 2. 4大AIの役割分担と全体サイクル
- **Gemini（リサーチエンジニア）**: 最新トレンド、公式仕様の正確な調査、難解なエラーログの解析。
- **ChatGPT（シニアソフトウェアアーキテクト）**: 要件定義、設計、技術選定の主導、およびCursor向け実装指示書の作成（コード記述禁止）。
- **Cursor（精密実装エンジニア）**: 指示書に基づいた最小限のクリーンコード実装と、開発者署名の強制（`.cursorrules`, `SKILL.md`）。
- **Claude（厳格なコードレビュアー・セキュリティエンジニア）**: 実装成果物と設計書の不整合、バグ、セキュリティ脆弱性の最終査読（迎合の禁止）。

## 4. 懸念点と運用上の注意事項
- **高いオーバーヘッド（手間）:** 複数のAIを往復するため開発のスピードは落ちます。「スピードより品質と安全、誠実なプロセス」を徹底する意志が必要です。（急がば回れを意識してください）
- **伝言ゲームのリスク:** Claudeの指摘を正しくChatGPTにフィードバックできないと、設計が先祖返りします。チャットが長大化しコンテキスト迷走の予兆が見られた際は、引き継ぎ精度を保ち先祖返りを防ぐため、所定の「移行テンプレート」の使用を【強く推奨します】。
- **パス規律:** バックアップ環境およびテスト環境のパスは、`COMMON_POLICY.md` に記載の規律を遵守すること。（Owner環境の例は同ファイルを参照。第三者は各自の環境に応じて同等の領域を定義すること）

## 5. 現場で生き抜くためのAI統治マニュアル
本章は、開発現場において生成AIを活用し、システムの品質と自身の身を守るための運用ガイドである。

### 5.1. Cursorの設定ファイル（.cursorrules / .mdc）を英文で記述する理由
本フレームワークでは、AIの行動を縛るルールを英語で記述している。これは保証ではなく、次の経験則に基づく。

- 英語の方が短く記述できる場合がある。
- 利用するモデルや指示内容によって効果は異なる。
- 英語記述だけで指示遵守を保証するものではない。

### 5.2. AIの設定を「PC全体」ではなく「Gitリポジトリ単位」に隔離する理由
CursorにはPC全体に適用する共通ルール（User Rules）も存在するが、実務では「Gitリポジトリ（フォルダ）ごと」に個別の設定ファイルを配置・管理（SCAO構成）することを推奨する。

**プロジェクトごとの仕様・規律の汚染防止:** 開発現場ごとに技術スタックやコーディング規約は異なる。PC全体に固定ルールを縛ると、別プロジェクトを開いた際にAIが文脈を混同し、ソースコードや署名コメントを意図せず汚染するリスクがある。

**チーム開発におけるガバナンス共有:** 設定ファイルをリポジトリ内に内包してGit管理しておくと、他のメンバーが `git clone` してCursorで開いた際、プロジェクトに合わせたルールが共有される。個人のローカル設定に依存させないことが、チーム開発の基本である。

### 5.3. 「パーソナライズ設定」 vs 「添付情報（Knowledge）」の使い分け
アカウント全体の共通設定に長大な仕様や詳細なペルソナを書き込むと、会話メモリ（コンテキスト）が初期段階から圧迫され、別業務の文脈にも干渉しやすくなる。

SCAOでは、共通設定には `COMMON_POLICY.md` の要約のみを数行で置き、詳細な仕様や各AIのPersona.mdはプロジェクトの添付資料（Knowledge）として隔離する。

- PersonaやKnowledgeは参照精度を高めるために有効である。
- 常に全文が完全参照されることを保証するものではない。
- 重要ルールは短い入口指示にも要約して残す。

### 5.4 Geminiによる最新情報調査の運用

Geminiは、最新技術、製品仕様、公式ドキュメント、複雑なエラーログの調査を担当する。

ただし、GemへPersonaを登録しただけで、外部検索が毎回必ず実行されるとは限らない。

最新性が必要な調査では、`0_research_gemini/GEMINI_RESEARCH_PROTOCOL.md` に定義した検索要求トリガーを使用する。

調査完了の判定では、公式URL、確認日、確定ファクト、過去情報との差分、未確認事項を人間が確認する。

### 5.5 自律型コーディングツールの比較

SCAOでは、標準実装担当としてCursorを採用する。

Cursor以外の自律型コーディングツールについては、Cursorと完全に同一の製品として扱わず、プロジェクト、チーム構成、セキュリティ要件、料金体系、利用環境、承認フローに応じて、代替候補または併用候補として評価する。

2026年時点の比較資料は、`docs/research/AGENTIC_CODING_TOOLS_2026.md` を参照する。
